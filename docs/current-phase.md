## 如何更新进度

本文件只做**快照**（当前在哪一阶段、链接指向哪）。详细任务、验收、交付记录写在 [`docs/v0.1/`](./v0.1/README.md) 各 `phase-N.md` 里。

### 开始一个新阶段

1. 更新本文顶部：**最后更新**、**当前**（例如 `阶段 1 — …（进行中）`）
2. 本文表格：将该阶段状态改为 `🔲 进行中`，并加上 `← **当前**` 标记；上一阶段若已完成保持 `✅`
3. 同步 [`v0.1/README.md`](./v0.1/README.md) 表格中对应阶段状态
4. 在 [`v0.1/phase-N.md`](./v0.1/phase-N.md) 顶部把 **状态** 改为 `🔲 进行中`
5. **下一步** 改链到当前 `phase-N.md`

### 完成一个阶段

1. 在 [`v0.1/phase-N.md`](./v0.1/phase-N.md) 按上文 **[阶段文档写法](#阶段文档写法供-ai)** 补全文档（含验收命令与真实结果），顶部 **状态** 改为 `✅ 已完成`
2. **提交代码**（阶段交付 commit），记下 `git rev-parse --short HEAD`
3. 在 [`v0.1/phase-N.md`](./v0.1/phase-N.md) 顶部附上 **Commit** id，**单独再 commit 一次文档**（避免 amend 导致 hash 与文档不一致）
4. 同步 [`v0.1/README.md`](./v0.1/README.md) 该阶段为 `✅ 已完成`
5. 更新本文顶部 **当前** 为下一阶段；表格中完成项改 `✅`，下一阶段标 `← **当前**`
6. **下一步** 改链到下一阶段文档
7. 若 v0.1 全部完成，**当前** 可写 `v0.1 已交付`，并更新 [roadmap.md](./roadmap.md) 版本状态

> **Commit** 记录的是**阶段交付 commit**（含代码与测试），不是后续只改文档的 commit。

### 阶段内部分完成

- **不**改本文阶段编号；只在对应 `phase-N.md` 里勾选任务清单 `[x]`
- 有阻塞时在本文 **阻塞项** 写一行，解除后改回「无」

## 阶段文档写法（供 AI）

各 `phase-N.md` 是**该阶段的交付说明**，完成后应像 [phase-2.md](./v0.1/phase-2.md)、[phase-3.md](./v0.1/phase-3.md) 一样可复盘，**不要**只留任务勾选或一行验收摘要。

### 推荐结构（按顺序，可略增减）

| 章节                | 写什么                                                                                                                               |
| ------------------- | ------------------------------------------------------------------------------------------------------------------------------------ |
| **文首元数据**      | 状态、完成日期、Commit（完成后填）、预估周期                                                                                         |
| **目标**            | 1 段话：本阶段要打通什么能力                                                                                                         |
| **任务清单**        | `- [x]` 勾选列表，与计划对齐                                                                                                         |
| **设计决策**        | 表格：关键选型与理由（ORM、鉴权、存储等）                                                                                            |
| **实际产出**        | **主体**：按包/子系统分节（`shared` / `core` / `server` / `cli` / `web`），写清新增文件、API、命令、配置；可用表格，避免粘贴大段代码 |
| **验收**            | 可复制的命令 + 预期结果（测试数、curl 示例）                                                                                         |
| **未做 / 留到后续** | 表格：刻意不做的事项 + 落在哪一阶段                                                                                                  |
| **下一阶段**        | 一句话 + 链接                                                                                                                        |

### 篇幅与风格

- **中等篇幅**：比 phase-0/1 的完成记录详细，比产品 spec 短；读者应能回答「改了哪些包、加了哪些入口、怎么验」。
- **完成时**：在文首标 `✅ 已完成` 和 **Commit**；`验收` 写真实跑过的命令与结果（可写在验收章节内，**不必**单独再开冗长的「完成记录」节，除非有额外交付说明）。
- **进行中**：可先有「设计决策」和「实际产出」草稿，验收节可先写命令模板，通过后补结果。

### 勿写入 phase-N.md

- 与实现无关的讨论过程、排错日志
- 密钥、token、真实 `.env` 内容
- 重复 `architecture.md` 全文；只写**本阶段增量**
